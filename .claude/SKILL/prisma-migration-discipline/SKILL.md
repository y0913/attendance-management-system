# Prisma Migration Discipline

Prisma スキーマ変更とマイグレーション運用のガード。雑なマイグレーションは即座にデータ損失につながるため、毎回必ず通す。

## 発動条件

- `schema.prisma` を編集しようとしている
- `npx prisma migrate dev`, `npx prisma migrate deploy`, `npx prisma db push` のいずれかを実行しようとしている
- 「マイグレーション作って」「DB スキーマ変えて」「テーブル追加して」「カラム追加して」「リレーション追加して」などの指示
- 「DB リセットして」のような破壊的指示

## スキーマ編集前のチェック

### Step 1: 既存スキーマと意図の確認

`schema.prisma` を編集する前に、現在のスキーマを `view` し、変更の意図を整理する：

- 何を追加するのか（モデル / フィールド / リレーション / インデックス）
- 何を変更するのか
- 何を削除するのか ← **特に注意**

### Step 2: 破壊的変更の検出

以下は破壊的変更。検出したら **必ずユーザーに警告して確認を取る**：

| 変更 | リスク |
|---|---|
| カラム削除 | データ消失 |
| カラム名変更 | データ消失（rename ではなく drop + add 扱い） |
| カラムの型変更 | データ消失またはキャスト失敗 |
| `?` を外して必須化 | 既存 NULL データで失敗 |
| `@unique` 追加 | 既存重複データで失敗 |
| enum 値の削除・リネーム | 既存データで失敗 |
| モデル削除 | データ消失 |
| リレーション削除 | 関連データの孤立 |

検出時の対応：
1. 何が破壊的かを具体的に説明
2. データ移行が必要か確認
3. 必要なら **段階的マイグレーション** を提案（追加 → データ移行 → 削除の3段階）

### Step 3: 命名・設計規約

- モデル名は **PascalCase 単数形**（`User` not `users` not `Users`）
- フィールド名は **camelCase**（`createdAt` not `created_at`）
- DB のテーブル/カラム名は `@@map` / `@map` で **snake_case** にマッピング
- 主キーは `id` で `String @id @default(cuid())` か `@default(uuid())`
- タイムスタンプは `createdAt DateTime @default(now())`, `updatedAt DateTime @updatedAt`
- 論理削除フラグより `deletedAt DateTime?` のソフトデリートを推奨
- リレーションは **双方向定義**（片側だけだと型推論が弱くなる）
- よく検索されるカラムには `@@index([fieldName])` を付与
- 複合ユニークは `@@unique([field1, field2])`
- enum は Prisma の `enum` を使う（文字列リテラル直書きは避ける）

### Step 4: フォーマット

スキーマ編集後は必ず実行：

```bash
npx prisma format
```

## マイグレーション実行

### 開発環境（migrate dev）

```bash
npx prisma migrate dev --name <descriptive_name>
```

#### マイグレーション名の規約

- **snake_case**
- **動詞始まり**で何をしたか表す
- 例: `add_users_table`, `add_email_to_users`, `rename_user_status`, `drop_legacy_table`
- 禁止例: `update`, `fix`, `migration1`, `temp`

### 本番環境（migrate deploy）

```bash
npx prisma migrate deploy
```

実行前に **必ず確認**：
1. ステージング環境で同じマイグレーションを通したか
2. 破壊的変更を含むマイグレーションか
3. バックアップは取られているか
4. ロールバック手順は決まっているか

ユーザーが上記を確認していない様子なら、実行前に必ず聞く。

### db push（プロトタイピング専用）

`prisma db push` は **マイグレーションファイルを作らずに DB を直接書き換える**。本番では絶対に使わない。開発初期のスキーマ試行錯誤フェーズのみ許可。使う場合は明示的に「これはプロトタイピング用途で本番では使わない」と確認する。

## 既存マイグレーションの保護

**既にコミット済みのマイグレーションファイルは絶対に編集しない**。チームメンバーや本番 DB との不整合を引き起こす。

### よくある間違いと正しい対応

| やりたいこと | NG な対応 | 正しい対応 |
|---|---|---|
| 過去のマイグレーションのミスを直したい | 既存マイグレーションファイルを編集 | 新しいマイグレーションを追加して修正 |
| マイグレーションを取り消したい | ファイル削除 | 取り消しのマイグレーションを新規作成 |
| ローカルでスキーマがおかしくなった | 既存マイグレーションを書き換える | `migrate reset`（**開発DBのみ**）または手動で DB 状態を直す |

### migrate reset の扱い

`prisma migrate reset` は **DB を空にして全マイグレーションを再実行する**。実行前に：

1. 環境が開発か明示確認（`DATABASE_URL` を見て本番でないか確認）
2. シードデータの有無を確認
3. ユーザーの明示的な承認を取得

本番 DB に対して実行するのは厳禁。

## 失敗時の対応

スキーマ変更や migrate コマンドが失敗した場合：

1. **エラー内容を正確に報告**（出力をそのまま見せる）
2. 原因仮説を提示
3. ユーザーの判断を仰ぐ
4. 勝手に `migrate reset` などの破壊的リカバリーをしない

## NG な振る舞い

- カラム削除を「軽微な変更」として警告なしに進める
- 既存マイグレーションファイルを編集
- マイグレーション名を雑につける（`update`, `fix` など）
- `db push` を本番環境で使う
- スキーマ編集後に `prisma format` をかけない
- リレーション片側だけ書いて済ませる
- 本番 DB に対してデータ移行手順なしで破壊的マイグレーションを流す
